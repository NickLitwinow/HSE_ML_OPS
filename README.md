## Описание проекта
**Система оценки кредитных рисков с автоматизированным MLOps пайплайном**

Проект представляет собой production-ready сервис для скоринга заемщиков и предсказания вероятности дефолта (Credit Default Prediction). Решение направлено на автоматизацию процесса принятия кредитных решений, минимизацию финансовых рисков банка и сокращение времени обработки заявок.

Основная цель — разработать отказоустойчивую систему, которая не только выдает точные прогнозы, но и обеспечивает полный цикл DataOps/MLOps практик: от версионирования сырых данных до мониторинга модели в эксплуатации.

## Технологический стек и MLOps практики
Решение построено на современном стеке технологий с упором на воспроизводимость и автоматизацию:

*   **ML Core:** Разработка моделей классификации (XGBoost/LightGBM/CatBoost) на табличных данных.
*   **DataOps:** Версионирование данных и пайплайнов обработки с помощью **DVC**. Гарантия воспроизводимости экспериментов на любых исторических срезах данных.
*   **Experiment Tracking:** Логирование метрик, параметров и артефактов обучения через **MLflow**. Сравнение экспериментов и выбор лучшей модели.
*   **CI/CD:** Автоматизированные конвейеры **GitHub Actions** для линтинга кода, запуска тестов (pytest) и сборки Docker-образов.
*   **Containerization:** Полная изоляция окружения с использованием **Docker**.
*   **Serving:** Высокопроизводительный REST API на базе **FastAPI** с валидацией входных данных (Pydantic) и документацией (Swagger UI).
*   **Monitoring:** (В планах) Отслеживание качества данных и дрейфа модели (Model Drift).

## Структура репозитория
- `src/`: Исходный код сервиса и пайплайнов обучения.
- `notebooks/`: Jupyter ноутбуки для EDA и прототипирования.
- `tests/`: Модульные и интеграционные тесты.
- `.github/`: Конфигурации CI/CD.
- `data/`: Локальное хранилище данных (под управлением DVC).
- `docker-compose.yml`: Конфигурация для локального запуска сервисов.

## Стратегия ветвления (Branching Strategy)
В проекте используется **GitHub Flow**:
1.  **main**: Основная ветка, содержащая всегда рабочий, протестированный код (production-ready).
2.  **feat/name**: Ветки для разработки новых функций (feature branches). Создаются от `main`.
3.  **fix/name**: Ветки для исправления ошибок.
4.  **Pull Requests**: Все изменения попадают в `main` только через Pull Request после прохождения тестов и Code Review.
